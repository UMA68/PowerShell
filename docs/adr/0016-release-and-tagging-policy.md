# ADR-0016: リリースタグ付与とRelease Notes運用基準を固定する
## Status
Accepted
## Context
本リポジトリでは、タグを切るタイミングとRelease Notesの書き方が運用者の判断に依存しやすく、「毎回タグを付けるのか」「どの変更を利用者に案内すべきか」が一貫しないリスクがあった。結果として、利用者が更新判断を行うための情報が不足したり、逆に内部差分の羅列で判断しづらくなったりする課題があった。

本方針は、個人運用およびPublic初期フェーズを前提としたものであり、
利用者や運用体制の変化に応じて見直す可能性がある。

タグとRelease Notesは、開発者向けの作業記録ではなく、利用者に対して「この版を受け取る意味」を示すための公開契約として扱う必要がある。そのため、タグの意味、付与基準、SemVerの実務判断、Release Notesの最小構成を明文化して固定する判断が必要である。
## Decision
- タグは変更履歴の全記録としては扱わず、人に差し出せる節目を固定する識別子として扱う。
- Public初公開のタイミングで `v1.0.0` を付与する。
- すべての変更ではタグを切らない。タグ対象は `main` ブランチ上の安定点のみとする。
- タグを切る基準は、利用者向け説明（README/Release Notes）を書ける変更、利用者が更新判断をする必要がある変更、外部仕様・振る舞い・期待結果に影響がある変更とする。
- 内部リファクターのみ、テストのみ、CIや補助スクリプト調整のみの変更は、原則としてタグを切らない。
- バージョン運用はSemVer実務ルールとして、PATCHを利用者影響のある不具合修正、MINORを後方互換を保った機能追加、MAJORを既存利用者に修正を要求する変更に対応させる。
- バージョン判断に迷った場合は、過剰な版上げを避けるため「上げない」または「MINOR」を選択する。
- Release Notesは差分説明ではなく、更新判断のための案内板として扱う。
- Release Notesには次の3点のみを書く。1) 何が変わったか（Added / Fixed / Changed）、2) 誰に影響があるか、3) 更新する理由があるか。
- Release Notesは次の最小テンプレを使用する。

```markdown
## vX.Y.Z
### Added / Fixed / Changed
- 利用者視点での変更点（1〜3行）
### Impact
- 既存の使い方は変わるか／変わらないか
- 更新推奨／必須かどうか
```

- Release Notesには内部実装の詳細やgit logの羅列を書かない。
## Consequences
- 👍 良い点
タグの意味が「公開節目」に固定され、利用者がバージョンを見て更新判断しやすくなる。どの変更を公開対象にするかの判断基準が統一され、運用の一貫性が高まる。Release Notesの記載内容が利用者視点に限定され、情報密度と可読性が向上する。
- 👎 悪い点・割り切り
内部改善の進捗はタグから直接追いにくくなり、詳細把握には別の記録を参照する前提になる。「迷ったら上げない」運用により、公開タイミングが慎重寄りになり、短期的な版数の増加は抑制される。安定点のみをタグ対象とするため、途中段階の成果を即時に配布する用途には適さない。

# ADR-0013: 定期（四半期）CIを環境変化検知専用とし、人間判断を前提とする

## Status

Accepted

## Context

本PowerShellリポジトリでは、CIを「mainブランチの健全性を守るための品質ゲート」と位置づけている。  
push / PR時に実行される常時CIは、プロダクトコードに対してのみ適用され、壊れたものをmainに入れないことを目的とする。

一方で、PowerShellランタイムやGitHub Actionsランナーは、リポジトリ側の変更とは無関係に更新される。  
これらの外部要因は、コード変更がなくてもCIを失敗させうる。

これらの更新影響を常時CIに含めると、品質ゲートとしてのCIがノイズに埋もれ、CIの信頼性と運用判断を損なうリスクがある。  
そのため、環境変化の検知と、コード品質の保証を明確に分離する必要があった。

## Decision

- PowerShellランタイム更新およびGitHub Actionsランナー更新の影響確認は、**定期（四半期）CI**として実施する。
- 定期CIの目的は「壊れたかどうかを自動で決定すること」ではなく、**環境変化を検知し、人間の判断を促す信号を出すこと**とする。
- 定期CIは `workflow_dispatch` または `schedule` により実行する。
- 定期CIが失敗した場合、GitHubからのメール通知を起点として、**人がログを確認し、対応要否を判断する**。
- 定期CIでは、自動修復・自動PR作成・自動マージなどは行わない。

## Consequences

### 👍 良い影響

- 常時CIの役割（品質ゲート）が明確に保たれる。
- ランタイムやランナー更新による一時的な失敗が、日常的な開発フローを阻害しない。
- CIを「決定装置」ではなく「判断を助ける信号機」として扱う文化が固定される。
- 将来、更新影響をどの粒度で扱うかを再検討する際の基準点が明確になる。

### 👎 トレードオフ

- 定期CIの失敗時には、人間がログを読み、判断するコストが発生する。
- 自動化による即時解決は行われないため、対応は人の意思決定に依存する。

## Alternatives Considered

### ランタイム・ランナー更新検知を常時 CI に含める

- 更新頻度や内容によってCIが不安定になり、品質ゲートとしての意味が薄れる。
- ノイズが増え、真に壊れた変更の検知が遅れるリスクがあるため却下した。

### 定期 CI の失敗を自動修復・自動 PR で対応する

- 原因切り分けが困難になり、更新影響に対する理解が浅くなる。
- 「なぜ壊れたのか」を学習する機会を失うため却下した。


## Notes

- 定期CIの具体的な運用フローや判断基準は、CI運用Playbookに委譲する。
- 本ADRは、将来、定期CIの頻度変更（月次・年次）や自動化範囲を見直す際の判断基準として参照されることを意図している。

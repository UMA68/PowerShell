# ADR-0015: ドキュメント配置・ディレクトリ構成方針

## Status

Accepted（2026-03-03 JST）

## Context

本リポジトリでは、スクリプト実行物、設計判断（ADR）、人向けドキュメント、プロンプトが同一階層や近接した場所に混在しやすく、目的別の探索に時間がかかる状態が発生していた。とくに、将来の保守時に「どこへ何を書くべきか」の判断が都度必要になり、運用の迷いとレビュー負荷の増加につながる課題があった。

一方で、本リポジトリの基本方針としてGitHubをSingle Source of Truthとし、設計判断はADRに記録することが定義されている。したがって、構成整理においては正本の所在を明確化しつつ、既存リンクや既存導線の互換性を不必要に壊さないことが必要である。

今回の整理は、実行コード・CI・Testsの挙動を変更しないことを前提に、移動と最小限のMarkdown追加に限定して実施された。

## Decision

ドキュメントは `docs/` 配下へ役割別に配置し、ADRの正本（canonical）は `docs/adr/` とする。構成は以下を基準とする。

```text
docs/
  adr/
    9000-meta/
  guides/
  prompts/
  runbooks/
    index.md
adr/（互換入口、README のみ）
```

旧 `adr/` は互換入口として維持するが、配置するのは `README.md` のみとする。`adr/` 側に個別ADRのリダイレクトファイルを大量作成する運用は採用しない。外部直リンクの実害が確認された場合のみ、例外的に最小本数で個別導線を追加する。

実行系フォルダー配下の `Readme.md` は移動しない。実体を複製せず、`docs/runbooks/index.md` をリンク集として運用する。これにより、相対リンク崩壊リスクと重複管理を回避する。

今後の構成整理は「最小主義（移動＋最小のMarkdown追加のみ）」を原則とし、ドキュメント都合で実行コード・CI・Testsの挙動を変更しない。

## Consequences

この決定により、目的別の探索性と可読性が向上し、ADR・ガイド・プロンプト・ランブック導線の所在が一貫する。結果として、日常運用時の判断コストが下がり、将来の自分を含む保守者が迷いにくくなる。

一方で、`docs/` 配下の役割分担を最初に理解するための学習コストは発生する。また、旧 `adr/` が入口として残るため、短期的には「正本は `docs/adr`、旧入口は `adr/README.md`」という二層導線を意識する必要がある。さらに、個別リダイレクトを量産しない方針により、古い直リンクの一部は入口経由でたどる運用が前提となる。

## Guardrails

- 新規ADRは必ず `docs/adr/` に作成する。
- `adr/` にはADR本文を置かず、`README.md` のみを維持する。
- `docs/runbooks/` はリンク集であり、実体の複製や移動先として使わない。
- ドキュメント整理を理由に、実行コード・CI・Testsの挙動を変更しない。
- 構成変更時は「移動＋最小限のMarkdown追加」の範囲に抑える。

# ADR-0004: パフォーマンステストを CI の Unit / Coverage から除外する

## Status
Accepted

## Context
本リポジトリでは Pester を用いて Unit テストおよび
コードカバレッジ測定を GitHub Actions 上で実行している。

しかし、以下の問題が発生した：

- Import-YamlConfig.Tests.ps1 に含まれる
  「大規模 YAML ファイル（多数のキーと配列）」のテストが、
  CI 環境では実行時間のばらつきにより失敗することがある
- このテストは **性能特性の検証** が目的であり、
  機能的な正否（Unit テスト）とは性質が異なる
- パフォーマンステストを Unit / Coverage と同時に実行すると、
  CI の安定性が損なわれる
- 結果として、テスト内容が正しくても
  CI が失敗する状況が発生した

一方で、パフォーマンステスト自体は有用であり、
完全に削除すべきものではない。

## Decision
以下の設計方針を採用する：

1. **パフォーマンステストには `Performance` タグを付与する**
2. CI 上の Unit テストおよびコードカバレッジ測定では  
   `Performance` タグを **ExcludeTag として除外**する
3. パフォーマンステストは以下の用途で実行する：
   - ローカル環境での手動実行
   - 将来的な夜間ジョブや専用ワークフロー
4. CI（Unit / Coverage）は  
   **機能の正当性と回帰検出に専念**させる

これにより、テストの意味論を明確に分離する。

## Consequences

### 👍 良い点
- CI の安定性が大きく向上する
- Unit テストと性能検証の責務が明確になる
- GitHub Actions の実行時間と失敗率が低下する
- 「CI が赤い = 機能が壊れている」という解釈が成立する

### 👎 悪い点・割り切り
- CI だけを見ても性能劣化は検知できない
- パフォーマンステストは意識的に実行しないと放置され得る
- Coverage から性能コードパスが外れる

## Notes（個人用・将来削除可）
- パフォーマンステストは「品質保証」ではなく
  **設計健全性の確認手段**
- CI に混ぜるとノイズになるため、意図的に分離する
- 将来 CI を強化する場合は、
  本 ADR を前提に「Performance 専用ワークフロー」を検討する
